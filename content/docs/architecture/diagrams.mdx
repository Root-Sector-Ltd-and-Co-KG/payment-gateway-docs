## Reverse Proxy Routing

```mermaid
flowchart TB
    Internet["Incoming HTTPS request"]
    Caddy["Caddy"]

    Internet --> Caddy

    Caddy -->|ADMIN_FRONTEND_DOMAIN| AdminHost["Admin Frontend"]
    AdminHost -->|/api/*| AdminBE["Admin Backend"]
    AdminHost -->|other paths| AdminFE["Admin React app"]

    Caddy -->|ADMIN_BACKEND_DOMAIN /api/*| AdminBE
    Caddy -->|ADMIN_BACKEND_DOMAIN /v1/*| Rewrite["rewrite /v1/* to /api/v1/*"]
    Rewrite --> AdminBE

    Caddy -->|MAIN_FRONTEND_DOMAIN| MainFE["Main React checkout / portal"]
    Caddy -->|MAIN_BACKEND_DOMAIN /api/*, /hooks/*, /.well-known/*, /health| MainBE["Main Backend"]

    Caddy -->|unknown HTTPS host| CatchAll["custom checkout domain catch-all"]
    CatchAll -->|/api/*, /hooks/*, /.well-known/apple..., /health| MainBE
    CatchAll -->|other paths| MainFE

    Caddy -.->|on-demand TLS ask| Ask["Admin Backend\n/api/v1/domains/verify"]
    Caddy -.->|ACME HTTP challenge| ACME["Admin Backend\n/api/v1/domains/acme-challenge/{token}"]
```

Specific domain blocks are evaluated before the `:443` catch-all. The catch-all enables custom checkout hostnames after the Admin Backend approves them through the Caddy `ask` endpoint.

## Checkout Creation And Payment

```mermaid
sequenceDiagram
    participant Merchant as Merchant Backend
    participant Admin as Admin Backend
    participant Mongo as MongoDB
    participant MainFE as Main Frontend
    participant MainBE as Main Backend
    participant Provider as Payment Provider

    Merchant->>Admin: POST /api/v1/checkouts/{siteId}/create
    Admin->>Admin: API key auth + checkout:create scope
    Admin->>Mongo: Load site, provider, org settings
    Admin->>Provider: Optional provider setup, e.g. Stripe PaymentIntent
    Admin->>Mongo: Insert checkout session
    Admin-->>Merchant: paymentUrl, session IDs, short code

    Merchant-->>MainFE: Redirect customer to paymentUrl
    MainFE->>MainBE: GET /api/v1/checkouts/{sessionPublicId}
    MainBE->>Mongo: Read session and branding
    MainBE-->>MainFE: Session, provider, tax, UI data

    MainFE->>MainBE: POST /api/v1/transactions/create
    MainBE->>Provider: Update/create provider payment object
    MainBE->>Mongo: Insert transaction
    MainBE-->>MainFE: transactionId, status, redirect data
```

The merchant server creates sessions on the Admin Backend. Customers interact with the Main Frontend and Main Backend. Card data is handled by provider browser SDKs and provider APIs; the gateway stores normalized transaction metadata.

Checkout preview and invoice preview are convenience flows. The authoritative tax state is the snapshot stored on transactions, invoices, credit notes, refunds, and chargebacks, plus the canonical `tax_records` used by tax reports and exports.

## Provider Webhook To Side Effects

```mermaid
sequenceDiagram
    participant Provider as Payment Provider
    participant Caddy as Caddy
    participant MainBE as Main Backend
    participant Mongo as MongoDB
    participant Worker as Worker
    participant Merchant as Merchant IPN endpoint

    Provider->>Caddy: POST /hooks/{type}/{providerId}
    Caddy->>MainBE: Forward webhook
    MainBE->>Mongo: Load provider config and webhook secret
    MainBE->>MainBE: Verify signature and parse event
    MainBE->>Mongo: Update transaction with guarded status transition
    MainBE->>Mongo: Create invoice/tax side effects when configured
    MainBE->>Mongo: Enqueue pdf:generate / email:send / slack:send / sms:send as needed
    MainBE-->>Provider: 200 OK or validation error
    MainBE-->>Merchant: Signed IPN POST when checkout has ipnUrl
    Worker->>Mongo: Claim queued jobs
```

Provider webhook implementations live in the Main Backend provider package. Outbound merchant IPN payloads are signed with the site's webhook secret and record delivery outcomes on the transaction.

## Background Job And PDF Flow

```mermaid
sequenceDiagram
    participant Admin as Admin Backend
    participant MainBE as Main Backend
    participant Mongo as MongoDB jobs
    participant Worker as Main Worker
    participant Gotenberg as Gotenberg
    participant Storage as Temp cache or org cloud storage

    Admin->>Mongo: Enqueue pdf:generate when invoice is issued/regenerated
    MainBE->>Mongo: Enqueue pdf:generate from checkout/portal flows when needed
    MainBE->>Mongo: Enqueue email:send, slack:send, sms:send
    Worker->>Mongo: Dequeue next pending job
    Worker->>Gotenberg: Convert HTML to PDF
    Worker->>Storage: Store generated document
    Worker->>Mongo: Update invoice/credit note with pdfUrl and pdfGenAt
    Admin->>Worker: Proxy document download over HTTP
    MainBE->>Worker: Proxy portal document download over HTTP
```

The Worker validates the Gotenberg connection on startup when configured, serves cached documents on its own HTTP port, and reuses the same process for notification delivery jobs.

## Licensing Flow

```mermaid
sequenceDiagram
    participant AdminUI as Admin Frontend
    participant AdminBE as Admin Backend
    participant Mongo as Local MongoDB
    participant License as Vendor license service

    AdminUI->>AdminBE: POST /api/v1/system/licensing/activate
    AdminBE->>License: Activate license for this installation
    License-->>AdminBE: Signed license claims and installation state
    AdminBE->>Mongo: Store local license identity and claims
    AdminBE-->>AdminUI: Current runtime state

    AdminUI->>AdminBE: POST /api/v1/system/licensing/heartbeat
    AdminBE->>License: Sync entitlement, version, and usage counters
    License-->>AdminBE: Updated signed claims
    AdminBE->>Mongo: Refresh local license state
```

The Admin Backend and Main Backend enforce runtime state locally. If the remote service is temporarily unavailable, the installation continues to use the locally stored signed claims and heartbeat-age rules described on the licensing page.

## Transaction Lifecycle

```mermaid
stateDiagram-v2
    [*] --> initiated : Checkout transaction created
    initiated --> pending_provider : Redirect, bank, wire, or mandate flow
    initiated --> succeeded : Immediate provider success
    initiated --> failed : Provider rejection
    initiated --> canceled : Customer or timeout cancellation
    pending_provider --> succeeded : Provider webhook or polling confirms
    pending_provider --> failed : Provider webhook or polling fails
    pending_provider --> canceled : Cancellation or expiry
    succeeded --> refunded : Refund processed
    refunded --> [*]
    canceled --> [*]
    succeeded --> [*]
    failed --> [*]
```

## Invoice Lifecycle

```mermaid
stateDiagram-v2
    [*] --> draft : Created manually or from transaction
    draft --> issued : Issue action
    draft --> [*] : Deleted
    issued --> [*] : Legal record remains

    state issued {
        [*] --> open
        open --> partially_settled : Partial settlement
        partially_settled --> settled : Full settlement
        open --> settled : Payment confirmed
        settled --> partially_refunded : Partial refund plus credit note
        partially_refunded --> refunded : Full refund plus credit note
        settled --> credited : Credit note fully offsets invoice
        open --> voided_or_reversed : Formal void or reversal workflow
        partially_settled --> voided_or_reversed : Formal reversal workflow
        refunded --> [*]
        credited --> [*]
        voided_or_reversed --> [*]
    }
```

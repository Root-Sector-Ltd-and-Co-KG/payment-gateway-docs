# Architecture Diagrams

## System Overview

```mermaid
graph TB
    subgraph Internet
        Customer[Customer Browser]
        Merchant[Merchant Backend]
        Provider[Payment Provider\nStripe / GoCardless]
    end

    subgraph Gateway
        Caddy[Caddy Reverse Proxy\nTLS + Custom Domains]
        MF[Main Frontend\nCheckout UI]
        MB[Main Backend\nPayment API]
        Worker[PDF Worker]
        AF[Admin Frontend\nAdmin Panel]
        AB[Admin Backend\nManagement API]
    end

    subgraph Data
        Mongo[(MongoDB)]
        Redis[(Redis / Garnet)]
        Gotenberg[Gotenberg\nPDF Service]
    end

    Customer -->|Checkout URL| Caddy
    Merchant -->|Create Checkout API| Caddy
    Caddy --> MF
    Caddy --> MB
    Caddy --> AF
    Caddy --> AB

    MB <--> Mongo
    MB <--> Redis
    AB <--> Mongo
    AB <--> Redis

    MB -->|Webhook events| Provider
    Provider -->|Webhook IPN| MB

    AB --> Worker
    MB --> Worker
    Worker --> Gotenberg
    Worker --> Mongo

    Caddy -.->|ACME / TLS storage| Mongo
    Caddy -.->|on-demand TLS verify| AB
```

Which service receives traffic for a given hostname and path is determined by the default **`payment-gateway-reverseproxy/Caddyfile`**. The diagram **rolls four public hostnames into one Caddy box**: merchant **create-checkout** calls go to **`ADMIN_BACKEND_DOMAIN`**, the customer **checkout UI** to **`MAIN_FRONTEND_DOMAIN`**, checkout **runtime APIs** and **provider webhooks** to **`MAIN_BACKEND_DOMAIN`**, and the **Admin Panel** to **`ADMIN_FRONTEND_DOMAIN`** (with `/api/*` on that host proxied to the Admin Backend — see the Caddyfile). **Mongo Express**, the **`:443` catch-all** for custom checkout domains, and **registry / licensing** traffic are omitted here; see [Hostnames & DNS conventions](../deployment/domains).

The shipped Caddy image uses the **`storage mongodb`** block so certificate metadata is written to the same MongoDB deployment (a separate concern from application collections). For **on-demand TLS**, Caddy’s `ask` URL targets the Admin Backend **`/api/v1/domains/verify`**. **Both** backends proxy invoice PDF bytes from the **Worker** (admin downloads and customer portal flows).

Hosted reference names (`api.payment-gateway.app`, `webhook.payment-gateway.app`, etc.) and self-hosted examples are summarized in [Hostnames & DNS conventions](../deployment/domains). For **which host** to use for merchant server calls vs provider webhooks, follow [Server integration checklist](../getting-started/integration-checklist).

## Request Flow — Checkout Creation

```mermaid
sequenceDiagram
    participant M as Merchant Backend
    participant AB as Admin Backend
    participant DB as MongoDB
    participant MF as Main Frontend
    participant MB as Main Backend
    participant P as Payment Provider

    M->>AB: POST /api/v1/checkouts/{siteId}/create
    AB->>AB: Validate API key scope (checkout:create)
    AB->>DB: Create checkout session record
    AB-->>M: { paymentUrl, id, sessionPublicId, shortCode? }

    Note over M: Redirect customer to paymentUrl

    M->>MF: Customer opens checkout page
    MF->>MB: GET /api/v1/checkouts/{sessionPublicId}
    MB->>DB: Fetch session + branding
    MB-->>MF: Session data + provider config

    MF->>MF: Render payment form

    MF->>P: Customer submits card details
    P-->>MF: Payment token / redirect URL

    MF->>MB: POST /api/v1/transactions/create
    MB->>DB: Create transaction record (initiated)
    MB->>P: Create payment intent
    P-->>MB: Payment confirmation
    MB-->>MF: { transactionId, status, redirectUrl }
```

> [!NOTE]
> **TLS** terminates at **Caddy** (not shown as a separate hop). In production, the merchant server calls **`ADMIN_BACKEND_DOMAIN`** for `POST …/checkouts/…/create`; the customer opens **`paymentUrl`** on **`MAIN_FRONTEND_DOMAIN`**; the checkout app then calls **`MAIN_BACKEND_DOMAIN`** for `GET`/`POST /api/v1/...` as above. Paths match the Main Backend router under `/api/v1`.

## Request Flow — Webhook to Transaction Update

```mermaid
sequenceDiagram
    participant P as Payment Provider
    participant MB as Main Backend
    participant DB as MongoDB
    participant W as Your Application

    P->>MB: POST /hooks/stripe/{providerId}
    MB->>MB: Verify provider signature (e.g. Stripe-Signature)
    MB->>DB: Fetch provider + webhook secret
    MB->>MB: Parse event type (payment_intent.succeeded)
    MB->>DB: Update transaction status to succeeded
    MB->>DB: Create invoice if configured
    MB->>W: POST signed IPN to checkout ipnUrl (if set)
    W-->>MB: 200 OK
    MB-->>P: 200 OK
```

Provider callbacks use **`MAIN_BACKEND_DOMAIN`** with paths under **`/hooks/...`** (no `/api/v1` prefix). The outbound **IPN** to your application is optional and only runs when the checkout specifies an `ipnUrl`.

## Encryption Key Hierarchy

```mermaid
graph TD
    ENV["SYSTEM_ENCRYPTION_KEY\nenvironment variable"]

    ENV -->|encrypts credentials for| SKMSCreds["KMS Provider Credentials\nstored encrypted in MongoDB"]
    SKMSCreds -->|used by| SKMS["System KMS Provider\nAWS / Azure / GCP / Vault"]
    SKMS -->|manages| SDEK["System DEK\nData Encryption Key"]

    SDEK -->|encrypts| SysData["System-Level Data\nUser accounts\nOrganization settings\nSystem configuration"]

    ENV -->|encrypts credentials for| OKMSCreds["Org KMS Credentials\nper-org, stored encrypted"]
    OKMSCreds -->|used by| OKMS["Organization KMS Provider\nper-org, optional BYOK"]
    OKMS -->|manages| ODEK["Organization DEK\nper-org"]

    ODEK -->|encrypts| OrgData["Organization-Level Data\nTransaction records\nCustomer information\nPayment details"]
```

## Transaction Lifecycle

```mermaid
stateDiagram-v2
    [*] --> initiated : Customer submits checkout
    initiated --> pending_provider : Async payment GoCardless or Wire
    initiated --> succeeded : Card / wallet often immediate
    initiated --> failed : Payment rejected
    pending_provider --> succeeded : Provider webhook confirms
    pending_provider --> failed : Provider webhook fails
    succeeded --> refunded : Refund processed
    refunded --> [*]
    succeeded --> [*]
    failed --> [*]
```

## Invoice Lifecycle

```mermaid
stateDiagram-v2
    [*] --> draft : Created manually or from transaction
    draft --> issued : Issue action
    draft --> [*] : Deleted
    issued --> [*] : Remains legal record

    state issued {
        [*] --> open
        open --> partially_settled : Partial settlement
        partially_settled --> settled : Full settlement
        open --> settled : Payment confirmed
        settled --> partially_refunded : Partial refund plus credit note
        partially_refunded --> refunded : Full refund plus credit note
        settled --> credited : Credit note fully offsets invoice
        open --> voided_or_reversed : Formal void or reversal workflow
        partially_settled --> voided_or_reversed : Formal reversal workflow
        refunded --> [*]
        credited --> [*]
        voided_or_reversed --> [*]
    }
```

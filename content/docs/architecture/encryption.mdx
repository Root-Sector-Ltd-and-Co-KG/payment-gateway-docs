# Encryption Architecture

The Payment Gateway employs a two-tier encryption architecture that supports PCI-DSS and GDPR-oriented security controls. Sensitive data such as KMS provider credentials, organization settings, and transaction details is protected in transit and can be protected at rest through system and organization encryption configuration.

## Two-Tier Key Hierarchy

The core design principle is the isolation of **System Data** from tenant-specific **Organization Data**.

### 1. System Level Encryption
The system level manages overarching gateway settings, user accounts, and credentials for different KMS providers.

- **`SYSTEM_ENCRYPTION_KEY`**: A master symmetric key provided via environment variables. It is solely used to encrypt and decrypt the credentials required to connect to the various KMS providers (e.g., Azure Key Vault service principals or AWS IAM credentials).
- **System KMS Provider**: The actual Cloud Key Management Service (e.g., AWS KMS) configured for the entire gateway platform.
- **System DEK (Data Encryption Key)**: Managed and rotated by the System KMS. This DEK is used to encrypt system-level settings and administrative user data.

### 2. Organization Level Encryption
To ensure data isolation, every tenant (organization) has its own independent encryption boundary.

- **Organization KMS Provider**: Organizations can bring their own KMS (e.g., HashiCorp Vault, Azure Key Vault) or ride on the gateway's default KMS. The connection credentials for these organization-specific KMS are secured centrally by the `SYSTEM_ENCRYPTION_KEY`.
- **Organization DEK**: A unique Data Encryption Key generated for each organization. It encrypts all transaction data, customer identities, payment details, and business settings. Crucially, **one organization's data can never be decrypted using another organization's DEK**.

## Supported KMS Providers

The system integrates transparently with major enterprise key management services via a common `KMS Provider Interface`:

- **AWS KMS**: Full integration with region-specific configuration and IAM-based authentication.
- **Azure Key Vault**: Complete integration with tenant-specific configuration and Service Principal authentication.
- **Google Cloud KMS**: Project-specific configuration utilizing Service Account authentication.
- **HashiCorp Vault**: Custom mount points and Token-based authentication for on-premise or specialized cloud deployments.

## Performance and Caching

KMS calls are latency-sensitive and can become a bottleneck at higher volume.
To reduce repeated provider round-trips, the encryption layer supports scoped
DEK caching and cache configuration controls in encryption settings.

Exact cache behavior depends on your configured deployment and cache settings.

## Encryption in Notifications (SMTP & IPN)

When the system asynchronously generates email notifications or pushes Instant Payment Notifications (IPNs) to external systems, it must dynamically decrypt `encTypes.FieldEncrypted` data safely.

The `NotificationService` relies on an injected `EncryptionFactory` to perform contextual decryption:

1. **Email Notifications**: Before sending an invoice receipt, the system creates an *Organization-Scoped Encryption Context*. It decrypts the recipient's `Email` field (stored encrypted in MongoDB). Additionally, the configured outbound SMTP credentials (e.g., `FromEmail`, `Username`, `Password`) are dynamically decrypted on the fly before pushing to the SMTP relay.
2. **IPN Payloads**: When creating the JSON payload for a generic webhook or IPN, the system fetches the full transaction. It decrypts billing/shipping primitives (like `FirstName`, `LastName`, and `Address`) securely, injects them into the HTTP payload, and signs the request using the decrypted Webhook Secret or API Key.

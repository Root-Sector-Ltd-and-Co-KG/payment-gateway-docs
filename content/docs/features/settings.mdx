# Advanced Settings

The **Settings** abstraction allows administrators to fine-tune the behavior of their Organization.

## Notifications (Email, Slack, SMS)

Under `Settings > Notifications`, you define how the platform communicates with your team and your buyers.

- **SMTP Service**: Configure your outbound mail server credentials (host, port, TLS). The gateway decrypts these credentials entirely on-the-fly when sending emails.
- **Slack Webhooks**: Receive real-time channel alerts for specific `payment.succeeded` or `payment.failed` event types.
- **Templates**: Customize document, email, Slack, and SMS templates with the Designer or explicit Advanced Source mode.

## Templates

Under `Settings > Templates`, you can manage document templates and notification templates in one workspace.

- **Document templates**: PDF layouts for invoices, pro forma documents, credit notes, and receipts.
- **Notification templates**: Email, Slack, and SMS messages for payment, invoice, credit-note, and portal OTP events.

Designer templates store their editable source as `builderDefinition`: semantic controls for branding, document sections, layout, variables, and channel-specific message blocks. Rendered HTML, markdown, or text is generated by the backend and treated as a preview/export artifact. Technical users can explicitly convert a template to **Advanced Source** when they need to author document HTML, email HTML, Slack markdown, or SMS text directly; after conversion, Designer controls are disabled because arbitrary source cannot be safely converted back into semantic controls.

Document templates can be created from presets, duplicated from a system default, previewed through the PDF proof renderer, exported/imported as JSON packages, set as defaults, and assigned to invoice flows. Imports show a review step and then create a new draft. The document composer supports a sortable section outline, page regions, optional section visibility, custom text blocks, dividers, spacers, and grouped controls for document metadata, seller data, client billing and shipping addresses, supported payment data, line items, VAT breakdown rows, footer data, and custom fields. VAT breakdown variables include country-aware fields such as `CountryCode`, `CountryName`, `CountryLabel`, `DisplayLabel`, and `VATBase`.

Template presentation controls such as logo placement, logo visibility, typography, colors, page margins, section order, and PDF-safe layout live in **Settings > Templates**. Organization-level invoice defaults such as the uploaded logo asset, payment-link creation behavior, wire-transfer bank data, and the default footer note remain in **Settings > Invoicing** so multiple templates can reuse the same business data without duplicating it.

Imports always create a new non-default draft template. They never overwrite an existing default or published template. Published revisions keep their source mode, builder definition or advanced source, and generated artifacts locked for runtime reproducibility.

Invoicing settings can point to an organization-wide default PDF template, while sites can override that template for storefront-specific branding. Site overrides store a concrete template ID; standard and corrective invoice business types still render through the `invoice` document template type. Draft pro forma output uses the `proforma` template type, and credit notes and receipts have their own template defaults.

## Invoicing

Under `Settings > Invoicing`, organizations configure two independent document-number layouts: one for invoices and one for credit notes.

- Each layout controls prefix, optional middle text, suffix, start number, digit width, year basis, visible date detail, separator, and reset policy.
- Fiscal-year labels are derived by the backend from the issue date and the configured fiscal-year start month.
- Sites may override only the invoice and credit-note prefixes. They do not create their own sequence counters or fiscal/calendar rules.

The full API payload shape for these settings is documented in the Admin API reference (see **Organization Settings** on the [Admin API](../api-reference/admin-api) page).

## Data Retention & Geolocation

The platform includes organization-scoped retention controls to support your
data minimization and lifecycle policies.

### Data Retention

Under `Settings > Retention`, you can configure:

- **Transaction Data Retention**:
  - Enable or disable automatic transaction-data cleanup.
  - Set retention period in days (minimum `30`, maximum `2555` / ~7 years in UI).
- **Customer Data Retention** (per field category):
  - IP Addresses
  - Billing Address
  - Shipping Address
  - Customer Email
  - Transaction Items
  - Optional "retain country information" toggle

`0` can be used for immediate deletion on supported customer-data fields.

> [!IMPORTANT]
> Retention values should be aligned with your legal/accounting obligations.
> Some jurisdictions require minimum retention periods for financial records.

### Geolocation Rules

Under geolocation settings, you can configure provider-backed geo checks and
apply policy decisions in checkout flows (for example, restriction handling for
specific jurisdictions).

## The Tax Engine

The gateway provides an organization-scoped tax configuration that is reused by hosted checkout, transactions, draft invoices, recurring invoices, refunds, chargebacks, and reporting.

![Tax settings page showing calculation guidance, buyer-country detection guidance, and Settings, Rules, and Rates tabs](/screenshots/app/tax-settings.png)

1. **Seller registration**: Configure seller country, VAT/tax IDs, OSS/IOSS registration metadata, UK VAT settings, and small-business exemption references where applicable.
2. **Tax rates**: Define effective-dated percentages such as `DE_STANDARD`. Future rates can be added ahead of time and are resolved by the transaction or invoice tax-point timestamp.
3. **Tax rules**: Connect rates to seller country, buyer country, buyer type, supply type, priority, and tax treatment such as `standard`, `reverse_charge`, `exempt`, `zero_rated`, `oss`, `export`, or `uk_vat`.
4. **Calculation behavior**: Choose whether merchant prices normally include tax, the rounding method, and the tax recognition mode used for canonical tax records.
5. **VAT validation and OSS evidence**: Enable VIES-backed EU VAT ID validation and strict OSS evidence checks when you want checkout to require non-conflicting buyer-location evidence.

Checkout-specific display preferences, such as whether buyers see tax breakdowns or whether the VAT ID field uses `none`, `format`, or `vies` validation, live under **Settings > Checkout**. The runtime mode is still gated by **Settings > Tax**, so checkout cannot show VIES-backed treatment when tax settings do not allow it.

For EU-focused merchants, automatic tax calculation should normally be paired with **tax included in price** and OSS registration when the merchant is registered for OSS. The seller country is the authoritative origin country for new calculations. Physical goods and shipping lines use the shipping destination when available, while virtual/service lines use the billing country, so mixed baskets can intentionally choose different VAT rate codes per line.

Tax-rule rate selection depends on the treatment. `standard` and `uk_vat` rules use an explicit configured VAT rate code. `oss` rules resolve the destination VAT rate from the buyer country at calculation time. `reverse_charge`, `exempt`, `export`, and `zero_rated` rules do not require a configured rate code.

The **Rebuild Tax Records** action repairs canonical reporting records from stored source snapshots. It preserves issued invoice and credit-note history instead of applying current rules to old legal documents. Use it after migrations or data repair when Tax by country, Revenue by country, or VAT OSS needs complete canonical `tax_records`.

**Reporting and API:** Tax by country, revenue by country, and VAT OSS filings in the Admin Panel are backed by canonical **tax records** and report endpoints (see [Invoicing & Tax architecture](../architecture/invoicing-and-tax) and the **Tax Management** / **Tax records** tables in the [Admin API](../api-reference/admin-api)). Programmatic rate maintenance includes `POST /api/v1/tax-rates/replace-defaults` (scoped `tax_rate:execute`), which syncs managed official default VAT percentage rates while preserving custom rates.
